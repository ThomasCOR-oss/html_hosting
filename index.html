Points de contrôle
Phase 1 : Vérifications Préalables

1.1 Audit de l'existant

  • Inventaire des dépendances
npm list --depth=0 npm outdated
  • Analyse du code
    • Identifier les packages tiers et leur compatibilité
    • Recenser les custom libraries internes
    • Vérifier l'utilisation d'APIs dépréciées
  • Documentation de l'architecture actuelle
    • Structure des modules
    • Services et intercepteurs
    • Configuration du routing
    • Configuration de build

1.2 Vérifications techniques

  • Version Node.js : Vérifier la compatibilité (Angular 19 nécessite Node 18.19+)
  • Package manager : npm/yarn version
  • TypeScript version actuelle
  • Tests existants : S'assurer qu'ils passent avant migration

1.3 Préparation de l'environnement

  • Backup complet du projet
  • Création d'une branche dédiée `migration/angular-19`
  • Setup d'un environnement de test isolé
  • Documentation des configurations spécifiques (proxy, environnements, etc.)
Phase 2 : Migration Progressive

Angular 6 → 7

ng update @angular/cli@7 @angular/core@7
  • Bundle budgets : Nouvelle configuration dans angular.json
  • CLI Prompts : Nouvelle fonctionnalité
  • Virtual Scrolling : Vérifier si utilisé
  • Drag and Drop : Nouvelle API CDK
Validation : `ng build` sans erreurs, Tests unitaires passent, Application démarre correctement, Fonctionnalités critiques testées manuellement.

Angular 7 → 8

ng update @angular/cli@8 @angular/core@8
  • Dynamic imports pour lazy loading
  • Ivy renderer (optionnel en v8)
  • Web Workers support
  • Differential loading automatique
Actions spécifiques : Migrer les lazy routes avec `ng update`, Vérifier les polyfills, Tester sur différents navigateurs.

Angular 8 → 9

ng update @angular/cli@9 @angular/core@9
  • Ivy renderer par défaut
  • ModuleWithProviders typing
  • Lazy loading syntax obligatoire
  • i18n nouvelle approche
Migration Ivy : Activer Ivy si pas déjà fait, Vérifier les custom libraries, Tester les dynamic components, Valider les ViewChild/ContentChild queries.

Angular 9 → 10

ng update @angular/cli@10 @angular/core@10
  • Date Range Picker
  • New Default Browser Configuration
  • Angular Material mise à jour
  • New File Structure pour i18n

Angular 10 → 11

ng update @angular/cli@11 @angular/core@11
  • Hot Module Replacement support
  • Component Test Harnesses
  • Updated Font Icons
  • TypeScript 4.0 support

Angular 11 → 12

ng update @angular/cli@12 @angular/core@12
  • Ivy everywhere (View Engine déprécié)
  • Sass support pour inline styles
  • Webpack 5 support
  • Strict mode par défaut
Actions critiques : Migrer de View Engine vers Ivy complètement, Vérifier les custom webpack configs, Tester les builds de production.

Angular 12 → 13

ng update @angular/cli@13 @angular/core@13
  • View Engine complètement supprimé
  • Angular Package Format changes
  • Node.js 16 support
  • TypeScript 4.4 support
Migration spécifique : Vérifier toutes les libraries tierces, Mettre à jour les custom libraries, Tester les builds et le tree-shaking.

Angular 13 → 14

ng update @angular/cli@14 @angular/core@14
  • Standalone Components (optionnel)
  • Angular CLI Auto-completion
  • Optional injectors in Embedded Views
  • Extended developer diagnostics

Angular 14 → 15

ng update @angular/cli@15 @angular/core@15
  • Standalone APIs stables
  • Directive composition API
  • Image optimization directive
  • MDC-based Angular Material
ng update @angular/material@15

Angular 15 → 16

ng update @angular/cli@16 @angular/core@16
  • Signals (developer preview)
  • Required inputs
  • Router data as input
  • Non-destructive hydration

Angular 16 → 17

ng update @angular/cli@17 @angular/core@17
  • New Application Builder
  • New Lifecycle Hooks
  • View Transitions API
  • SSR improvements

Angular 17 → 18

ng update @angular/cli@18 @angular/core@18
  • Material 3 support
  • Control flow syntax stable
  • Event replay for SSR
  • Built-in control flow (@if, @for, @switch)

Angular 18 → 19

ng update @angular/cli@19 @angular/core@19
  • Signals stables
  • Standalone ng-new collection
  • Improved hydration
  • Performance optimizations

Avant de mettre à jour
Remplacez Renderer par Renderer2
Si vous utilisez HttpModule et le service Http, passez à HttpClientModule et au service HttpClient. HttpClient simplifie les ergonomies par défaut et prend désormais en charge les valeurs de retour typées et les intercepteurs.
Le support de ngModel avec les directives de formulaire réactives a été déprécié en v6 et supprimé en v7.
Supprimez les fonctionnalités dépréciées de RxJS 5 en utilisant les règles de mise à jour automatique rxjs-tslint.
Pour la plupart des applications, cela signifie exécuter les deux commandes :
npx rxjs-tslint rxjs-5-to-6-migrate -p src/tsconfig.app.json
Une fois que vous et toutes vos dépendances ont été mises à jour vers RxJS 6, supprimez rxjs-compat.
Mettez à jour vers la nouvelle version.
Si vous utilisez le service worker Angular, migrez les fichiers versionnés vers le tableau de fichiers.
Angular utilise maintenant TypeScript 3.1, lisez les notes de version pour les changements potentiels.
Angular a maintenant ajouté le support pour Node 10.
Mettez à jour vers v7 du framework et CLI en exécutant :
cmd /C "set "NG_DISABLE_VERSION_CHECK=1" && npx @angular/cli@7 update @angular/cli@7 @angular/core@7"
Mettez à jour Angular Material vers v7 :
cmd /C "set "NG_DISABLE_VERSION_CHECK=1" && npx @angular/cli@7 update @angular/material@7"
Si vous utilisez des tests de capture d'écran , vous devrez régénérer vos fichiers d'or pour les captures d'écran, car de nombreux ajustements visuels mineurs ont été apportés.
Arrêtez d'utiliser matRippleSpeedFactor et baseSpeedFactor pour les ripples, utilisez plutôt la configuration d'animation.
Mettez à jour vers la version 8 du framework et CLI :
cmd /C "set "NG_DISABLE_VERSION_CHECK=1" && npx @angular/cli@8 update @angular/cli@8 @angular/core@8"
Remplacez /deep/ par ::ng-deep dans vos styles.
Angular utilise maintenant TypeScript 3.4, lisez les notes de version pour les erreurs potentielles.
Assurez-vous d'utiliser Node 10 ou une version ultérieure.
La commande de build du CLI crée automatiquement une build moderne ES2015 avec des polyfills minimaux.
Lorsque vous utilisez de nouvelles versions du CLI, vous serez invité à partager vos données d'utilisation du CLI.
Si vous utilisez ViewChild ou ContentChild, nous mettons à jour la façon dont nous résolvons ces requêtes.
Mettez à jour Angular Material vers la version 8 :
cmd /C "set "NG_DISABLE_VERSION_CHECK=1" && npx @angular/cli@8 update @angular/material@8"
Au lieu d'importer depuis @angular/material, vous devez importer profondément depuis le composant spécifique.
Pour les modules chargés paresseusement via le routeur, assurez-vous d'utiliser des imports dynamiques.
Nous déprécions le support pour @angular/platform-webworker.
Nous avons changé du compilateur Sass natif au compilateur JavaScript.
Si vous construisez vos propres schématiques, elles étaient potentiellement asynchrones.
Assurez-vous d'utiliser Node 10.13 ou une version ultérieure.
Exécutez la commande suivante pour mettre à jour vers la dernière version 8.x :
cmd /C "set "NG_DISABLE_VERSION_CHECK=1" && npx @angular/cli@8 update @angular/cli@8 @angular/core@8"
Vous pouvez passer l'option --create-commits pour créer un commit git par migration individuelle.
Exécutez la commande suivante pour mettre à jour vers la version 9 d'Angular :
cmd /C "set "NG_DISABLE_VERSION_CHECK=1" && npx @angular/cli@9 update @angular/cli@9 @angular/core@9"
Votre projet a maintenant été mis à jour vers TypeScript 3.8.
Exécutez la commande suivante pour mettre à jour Angular Universal :
cmd /C "set "NG_DISABLE_VERSION_CHECK=1" && npx @angular/cli@9 update @nguniversal/hapi-engine@9"
Si votre projet dépend d'autres bibliothèques Angular, envisagez de les mettre à jour vers leur dernière version.
Lors de la mise à jour vers la version 9, votre projet a été transformé pour supprimer les appels d'API incompatibles ou dépréciés.
Les styles et classes CSS liés étaient appliqués avec une stratégie "dernier changement gagne".
Si vous êtes un auteur de bibliothèque et que vous aviez une méthode retournant ModuleWithProviders, vous devrez spécifier le type générique.
Le support pour le cadre de traçage web dans Angular a été déprécié.
Supprimez les drapeaux es5BrowserSupport dans votre angular.json et définissez votre cible sur es2015 dans votre tsconfig.json.
Si vous utilisez l'élément ngForm pour créer des formulaires Angular, utilisez plutôt ng-form.
Nous avons mis à jour le tsconfig.app.json pour limiter les fichiers compilés.
Avec Angular 9, Ivy est maintenant le moteur de rendu par défaut.
Si vous utilisez Angular Universal, plusieurs fichiers de sauvegarde seront créés.
Angular 9 a introduit une fonction globale $localize qui doit être chargée si vous dépendez de l'internationalisation d'Angular.
Dans vos projets d'application, vous pouvez supprimer les NgModules entryComponents.
Si vous utilisez TestBed.get, utilisez plutôt TestBed.inject.
Si vous utilisez le support i18n d'Angular, vous devrez commencer à utiliser @angular/localize.
Assurez-vous d'utiliser Node 12 ou une version ultérieure.
Exécutez la commande suivante pour mettre à jour vers la version 10 d'Angular :
npx @angular/cli@10 update @angular/core@10 @angular/cli@10
Exécutez la commande suivante pour mettre à jour Angular Material :
npx @angular/cli@10 update @angular/material@10
Les nouveaux projets utilisent le fichier .browserslistrc au lieu de browserslist.
Angular nécessite maintenant tslint v6, tslib v2 et TypeScript 3.9.
Arrêtez d'utiliser styleext ou spec dans vos schématiques Angular.
Dans la version 10, les classes qui utilisent des fonctionnalités Angular et n'ont pas de décorateur Angular ne sont plus prises en charge.
À partir d'Angular 9, l'application des décorateurs @Injectable pour DI est plus stricte.
Les paquets NPM d'Angular ne contiennent plus de commentaires jsdoc.
Si vous utilisez des formulaires Angular, les entrées de type number n'écoutent plus les événements de changement.
Pour la validation des formulaires Angular, les validateurs minLength et maxLength vérifient maintenant que la valeur du contrôle de formulaire a une propriété de longueur numérique.
Le format de paquet Angular a été mis à jour pour supprimer les formats esm5 et fesm5.
Les avertissements concernant les éléments inconnus sont maintenant enregistrés comme des erreurs.
Tout résolveur qui retourne EMPTY annulera la navigation.
Si vous utilisez le service worker Angular et dépendez de ressources avec des en-têtes Vary, ces en-têtes sont maintenant ignorés.
Vous pourriez voir des erreurs ExpressionChangedAfterItHasBeenChecked qui n'étaient pas détectées auparavant.
Si vous avez un binding de propriété tel que [val]=(observable | async).someProperty, cela ne déclenchera plus la détection des changements si la valeur de someProperty est identique à l'émission précédente.
Si vous utilisez formatDate() ou DatePipe avec des codes de format b ou B, la logique a été mise à jour.
Si vous utilisez le UrlMatcher, le type reflète maintenant qu'il pourrait toujours retourner null.
Pour plus de détails sur les dépréciations, les migrations automatisées et les changements, consultez le guide angular.io.
Pour les utilisateurs d'Angular Universal, si vous utilisez useAbsoluteUrl pour configurer platform-server, vous devez également spécifier baseUrl.
